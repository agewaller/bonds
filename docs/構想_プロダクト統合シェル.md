# 構想: プロダクト統合シェル LMS（一回ログイン・タブ横断・連携の一回化）

**呼称**: オーナーはプロダクト統合 LP を **LNS（ライフマネジメントシステム）** とも呼ぶ
（2026-07-21。既存リポジトリ agewaller/lms の Life Management System と同思想。表記ゆれ
LNS/LMS はどちらもこの統合シェル構想を指す）。

**起草**: 2026-07-21（オーナー依頼「LPから一回ログインすると上部に各プロダクトのタブが並び
自由に行き来。ユーザー毎にプロダクトを跨いでデータを一括共有。gmail/カレンダー/plaud などの
連携も一回で全プロダクトに効くように。可能ですか？」への回答の恒久記録）
**結論**: 可能。ただし一夜ではなく三段階。各段が単独で価値を出す。

## 事実（2026-07-21 時点の土台）

- bonds と cares は **Firebase プロジェクト共用済み**（care-14c31）＝この2つは既に「同じアカウント」
- zentrack・VM user-console も Firebase Auth だが **プロジェクトは env 設定**（コード変更なしで
  同一プロジェクトへ寄せられる）。VM はマルチテナント設定（VITE_USERS_TENANT_ID）のため
  テナントの扱いだけ設計が要る
- bonds には連携の器がすでにある: `google_connections`（OAuth 接続・暗号化）・毎時 sweep・
  取込ハブ（メール添付/ファイル/写真、デバイス非依存）・外部参照つき人物集約（integration hub）

## 三段階

### 第1段: 見た目の統合（安い・すぐ）
- 全プロダクトの Firebase プロジェクトを care-14c31 に統一（env 変更が主）
- 共通ヘッダー「プロダクトタブ」を各アプリ上部に（禅トラック/健康日記/bonds/GF/VM + LP へ戻る）。
  最初は単なる相互リンクでよい（SSO 完成前でも同じ Google アカウントなら再ログインは1クリック）
- LP の「Google アカウントでつながります」セクションがこの導線の入口になる

### 第2段: 連携の一回化（bonds を接続ハブに）
- Google（gmail/カレンダー/連絡先）・Plaud・録音デバイスの接続を **bonds の connections を正本**とし、
  他プロダクトはサーバ間 API（トークン非公開・データだけ渡す）で参照する
- 例: 禅トラックのカレンダー表示・cares の Plaud 取込は、bonds が取った接続を使う
  （ユーザーは一回つなげば全プロダクトで有効）
- 同意の粒度は「どのプロダクトに何を渡すか」を設定画面で見える化（データ主権原則）

### 第3段: データの一括共有（= パーソナルOS構想と同一）
- 人物・予定・健康・意識・資産の記録を、同意ゲート付きの共通文脈層で相互参照
- これは `docs/検討論点_責任を取るOS.md` の「本人の文脈と同意を預かる1つの層」の実装形。
  第2段までの配管がそのまま土台になる

## デバイス取り込み（優先順位と実装状況）

1. スマートリング（Oura API）— **実装済み（2026-07-21・格納は bonds＝共通取り込み基盤）**:
   `device_connections`（トークン暗号化）+ `health_metrics`（日次・payload 暗号化・冪等 upsert）。
   OAuth: `/api/devices/oura/auth-url` → callback（HMAC state）→ 設定ページ「からだの記録の
   デバイス連携」でつなぐ/同期/外す。毎時 sweep `admin/devices/sync-all`。読み手は
   `GET /api/health/metrics`（cares/LMS がここを読む）。env 未設定は「準備中」縮退。
   オーナー設定は OWNER-SETUP.md タスク7
2. マット型睡眠（Withings API）— **実装済み（同上・同じ基盤の provider=withings）**
3. 会話ウェアラブル（Omi 等）— **実装済み（2026-07-21）**: `POST /api/inbound/conversation`
   （共有シークレット認証・Omi の transcript_segments と素の text の両対応・Plaud と同じ
   録音メモ+タスク整理パイプラインへ合流・外部 id で冪等・AI 不可でもメモは残る）。
   加えて **手動の取り込み口**（2026-07-21）: `POST /api/relationship/voice-memos`＝どの
   レコーダー・文字起こしアプリのテキストでも、連絡帳の録音メモパネルに貼り付ければ同じ
   タスク整理に乗る（Plaud 非依存化）。
   本番の INBOUND_WEBHOOK_SECRET は既存の受信 webhook と共用＝新しいオーナー設定は不要。
   デバイス側（Omi アプリの webhook 先 URL 設定）だけがオーナー作業
4. CGM — 日本の入手性・規制の確認が先（要検証）

**cares/LMS 側の残作業**: cares の設定画面に「デバイス連携」への導線（bonds の設定ページへの
リンク or 埋め込み）と、`GET /api/health/metrics` を読んで健康日記のグラフに重ねる表示。
agewaller/cares がセッションに無いため未着手（「add agewaller/cares」で着手可能）。
