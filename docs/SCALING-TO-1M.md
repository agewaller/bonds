# 100万人が使える堅牢なシステムへの道筋（工数・技術・コスト）

作成: 2026-07-23（オーナー依頼「100万人が使える堅牢なシステムにするために必要なことを書き留めて」）
対象: bonds を中心に、健康日記（cares）・禅トラック・統合LP・VM を含むプロダクト群。

## 大原則

**いま100万人分を作らない。次の10倍だけを、後戻りできる形で作る。**
現在の利用者は数十人。ここで100万人向けの分散システムを組むのは、コストと複雑さで
プロダクトの進化を殺す（CLAUDE.md の「複雑化・退行の禁止」と同思想）。
そのかわり、各段階に「次の段階へ進む引き金（メトリクス）」を決めておき、
引き金が引かれたらその段の工事だけをやる。

## 現状の資産（書き直し不要。土台は正しい）

- **ステートレスな Cloud Run**（api / web）— 水平スケールの素地がある。サーバにセッションを持たない
- **ownerUid による全データのテナント分離** — マルチユーザー化の骨格が最初から入っている
- **アプリ層暗号化（AES-256-GCM）・レートリミッタ・SSRF 対策・三段認証** — セキュリティの底が固い
- **テストゲート（unit 421 / integration 252 / e2e 監査）と staging** — 変更を壊さず積める
- **プロンプト DB 駆動・AI 月次キャップ（ユーザー別）** — 変動費の制御装置が既にある

## 現状の単一障害点・ボトルネック（危険な順）

1. **DB が単一の Cloud SQL インスタンス**。全機能が一つの Postgres に依存。バックアップはあるが、
   接続数・CPU・ストレージのどれかが先に天井になる
2. **Prisma × Cloud Run の接続数爆発**。インスタンスが水平に増えると DB 接続数が掛け算で増え、
   数百人規模の同時利用で `too many connections` が起きうる（**最初に壊れるのはここ**）
3. **毎時 sweep が GitHub Actions cron 頼み**。バッチが「全ユーザーを1プロセスで順番に」処理する
   設計のため、ユーザー数に比例して1回の sweep が伸び、やがて1時間で終わらなくなる
4. **禅トラックが単一 EC2 + systemd**。再起動・障害・容量のすべてが手動。プロダクト群で唯一の
   ペット（手飼い）サーバ
5. **一覧系 API の全件取得**。連絡先など「ユーザーの全行を返す」エンドポイントは、1ユーザーの
   データ量が増えると応答が重くなる（ページネーション未導入の箇所がある）
6. **メール基盤の依存**（Resend 無料枠・アカウント審査リスクは 2026-07 に実体験済み）
7. **AI 変動費**。ユーザー課金額 < AI 原価の逆ざやは、規模が増えるほど致命傷になる

## 段階計画

### 段階A: 〜1,000人（いまやる。目安 2〜3週間の実装）

引き金: 課金ユーザーの受け入れ開始（もう引かれている）。

- **接続プール**: PgBouncer（Cloud SQL の組み込み or Cloud Run sidecar）を挟み、Prisma の
  connection_limit を明示。max instances × 接続数 < DB 上限を常に保つ
- **ページネーション**: 連絡先一覧・記録一覧など全件取得の API をカーソル方式に
- **インデックス点検**: ownerUid 複合索引の網羅、EXPLAIN での遅いクエリ洗い出し
- **エラートラッキング**: Sentry（無料枠）を api/web に。いまは「ユーザーが言ってくれたら気づく」
- **バックアップの復元試験**: 取っているバックアップから実際に復元できることを staging で年2回確認
- **死活監視と通知**: healthz の外形監視（UptimeRobot 等無料枠）→ オーナーに通知

費用: ほぼ増えない（月 +0〜1万円）。工数: 実装は Claude Code で可、レビューとデプロイ承認のみオーナー。

### 段階B: 〜3万人（目安 2〜3ヶ月）

引き金: DAU 500 超、または sweep 1回が15分超、または DB CPU 常時 50% 超。

- **ジョブ基盤の独立**: 毎時 sweep を GitHub Actions から **Cloud Scheduler + Cloud Tasks/Pub-Sub +
  ワーカー（Cloud Run jobs）** に移す。「全ユーザー一括」から「ユーザー単位のジョブに分割して
  並列処理・リトライ・冪等」に。ここが100万人対応の本丸its
- **読み取りレプリカ**: Cloud SQL にリードレプリカ1台。重い読み(一覧・集計)をレプリカへ
- **ユーザー別クォータの本格化**: AI キャップ（実装済み）に加え、API レート・ストレージ量・
  取込サイズの上限をプラン別に
- **メールを SES（または SendGrid 有償）へ**: 送信ドメイン専用化・ウォームアップ・バウンス/苦情の
  自動処理。Resend 無料枠依存から卒業
- **AI 原価設計**: 機能ごとにモデル階層を分ける（整理系は Haiku、評価系だけ上位モデル）、
  キャッシュ、バッチ化。**目標: アクティブユーザー1人あたり月 AI 原価 ¥50〜150 に収める**
- **禅トラックの移設**: EC2 → コンテナ化して Cloud Run + Cloud SQL（または bonds への機能統合）
- **可観測性**: 構造化ログの集約・p95 レイテンシと エラー率の SLO・アラート

費用: 月 20〜50万円（DB HA+レプリカ、ワーカー、メール、監視）。
工数: 実装 3〜4人月相当（Claude Code 主体で可）。**この段階から人間の運用当番（週次でアラートを
見る人）が1人必要**。

### 段階C: 〜30万人（目安 3〜6ヶ月）

引き金: DAU 5,000 超、または p95 > 1秒が常態化、またはレプリカ込みで DB CPU 70% 超。

- **DB の本格増強**: AlloyDB（Postgres 互換・読み書き性能数倍）へ移行、または大テーブル
  （contact_interactions / voice_memos / ai_usage_logs）の時系列パーティション化
- **多リージョン DR**: クロスリージョンレプリカ + 復旧手順書 + 年2回の DR 訓練
- **エッジ防御**: Cloud Armor（WAF・DDoS）、Cloud CDN で静的資産配信
- **セキュリティ外部監査**: ペネトレーションテスト（1回 100〜300万円）、鍵管理を KMS ローテーション
  運用に、脆弱性スキャンの CI 組み込み
- **サポート体制**: 問い合わせ窓口（ヘルプデスク SaaS）・FAQ・退会/データ削除のセルフサービス
- **法務**: 利用規約・プライバシーポリシーの弁護士レビュー、個人情報保護法の安全管理措置の文書化

費用: 月 100〜300万円（インフラ）+ 監査等の一時費用。
工数: 4〜6人月相当 + **専任 SRE 1名（または業務委託）**がここから必須。

### 段階D: 100万人（目安 6〜12ヶ月・段階Cの上に）

引き金: DAU 5万超。

- **データ層の最終形**: AlloyDB のスケールアップ + 必要ならユーザー ID によるシャーディング
  （ownerUid 分離が最初からあるため、シャーディングへの道は既に舗装されている）
- **チーム**: インフラ/SRE 2〜3名、セキュリティ担当1名、サポート数名（外部委託可）
- **コンプライアンス**: ISMS（ISO27001）取得を推奨（法人顧客・提携の信用の担保になる）
- **費用感（月次・概算）**:
  - インフラ（DB・コンピュート・CDN・監視）: 300〜800万円
  - メール（月数百万通）: 10〜50万円（SES 前提）
  - **AI: 最大の変動費**。アクティブ率20%・1人月 ¥100 なら 2,000万円/月 → **プラン価格が
    AI 原価の3倍以上になるよう機能の出し方を設計するのが生命線**
  - 人件費: 上記チームで月 500〜1,000万円

## いま決めておくべきこと（技術以外）

1. **単価設計**: AI 原価が乗る商売なので、「無料で何をどこまで出すか」が損益を決める。
   無料枠は AI を使わない機能（純粋関数系の提案・記録・カレンダー）中心に、AI 生成は有料プランへ
2. **独自ドメイン**: Google アプリ確認（OWNER-SETUP タスク6）・メール到達性・SSO 統合の
   すべての前提。早いほど安い
3. **データ削除とエクスポートのセルフサービス**（データ主権原則の延長）: 規模が増えてからの
   後付けは高くつく。設定ページに実装済みの書き出しに「アカウント削除」を足しておく

## 引き金メトリクスの観測方法

- DAU / 登録ユーザー数: 管理画面に週次表示（未実装・段階Aで追加）
- sweep 所要時間: Actions のログ（現状でも見られる）
- DB CPU / 接続数: Cloud Console の Cloud SQL モニタリング
- p95 レイテンシ / エラー率: Sentry + Cloud Monitoring（段階Aで導入）
