## 3. bonds を本番にデプロイする（1〜2時間・まとまった集中時間向け）

**なぜ**: いま bonds のデータ（28名・提供残高・あなたのカレンダー）は開発環境にあります。
本番に出して初めて、スマホからリンクを送る実運用ができます。

**準備は完了しています**: デプロイスクリプト（`infra/scripts/00〜07`）は実装済みで、
今日追加した機能の環境変数（OWNER_EMAIL・PUBLIC_WEB_ORIGIN・受信シークレット・
セッションCookie用サービスアカウント）も 07 に配線済み。**Cloud Shell で上から実行するだけ**です。

**手順（Google Cloud Shell で。https://shell.cloud.google.com を開く）**:

1. コードを取得（作業ブランチをそのまま使う。main への統合は別途）:
   ```
   git clone https://github.com/agewaller/bonds.git && cd bonds
   git checkout claude/self-validating-advisor-v0gsmc
   ```
2. 初回セットアップ（冪等・2回実行しても安全）:
   ```
   bash infra/scripts/00-first-time-setup.sh
   bash infra/scripts/01-create-secrets.sh
   ```
3. シークレットに値を投入（1行ずつ。BONDS_FIREBASE_SERVICE_ACCOUNT_JSON は
   Firebase Console → プロジェクト設定 → サービスアカウント → 新しい秘密鍵 で
   ダウンロードした JSON を1行にして貼る）:
   ```
   printf '%s' "$(openssl rand -hex 32)" | gcloud secrets versions add BONDS_DATA_ENCRYPTION_KEY --data-file=- --project=arctic-anvil-497002-q2
   printf '%s' "$(openssl rand -hex 32)" | gcloud secrets versions add BONDS_ADMIN_BREAKGLASS_TOKEN --data-file=- --project=arctic-anvil-497002-q2
   printf '%s' "$(openssl rand -hex 24)" | gcloud secrets versions add BONDS_INBOUND_WEBHOOK_SECRET --data-file=- --project=arctic-anvil-497002-q2
   printf '%s' "<DBパスワード任意>" | gcloud secrets versions add BONDS_DB_PASSWORD --data-file=- --project=arctic-anvil-497002-q2
   cat service-account.json | tr -d '\n' | gcloud secrets versions add BONDS_FIREBASE_SERVICE_ACCOUNT_JSON --data-file=- --project=arctic-anvil-497002-q2
   ```
   （メール送信のキーは項目3b のときでOK。未投入でも送信系だけ503になり他は動きます。
   ただし BONDS_FIREBASE_SERVICE_ACCOUNT_JSON は**必ず投入**——空のままだと Cloud Run が起動しません）
4. DB移行 → イメージ → デプロイ（NEXT_PUBLIC_* は cares と同じ Firebase の値）:
   ```
   bash infra/scripts/05-migrate-prod.sh
   NEXT_PUBLIC_FIREBASE_API_KEY=<値> NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN=<値> NEXT_PUBLIC_FIREBASE_PROJECT_ID=care-14c31 bash infra/scripts/06-build-push-images.sh
   bash infra/scripts/07-deploy-cloud-run.sh
   ```
5. 07 の最後に表示される web の URL を使って**もう一度 API だけ再デプロイ**
   （CORS と相手向けリンクの origin を web の実URLに合わせるため。画面の指示どおり）:
   ```
   bash infra/scripts/07-deploy-cloud-run.sh --only=api --web-url=<webのURL> --api-url=<apiのURL>
   ```
6. 動作確認 → シード投入（28名と提供残高が本番に入る。何度実行しても重複しません）:
   ```
   curl <apiのURL>/api/healthz
   API_BASE=<apiのURL> ADMIN_TOKEN=$(gcloud secrets versions access latest --secret=BONDS_ADMIN_BREAKGLASS_TOKEN --project=arctic-anvil-497002-q2) node scripts/seed-vm-offers.mjs
   ```

つまずいたらエラーメッセージをそのまま貼ってください。次のセッションの私が対応します。

## 3b. Resend の設定 — VM アウトリーチの送信を有効にする（10分・項目3の前後どちらでも）

**なぜ**: VM アウトリーチ（Tier2 46社・下書き46通はシード済み）を「山口揚平」名義で送り、
返信をあなたの Gmail に届けるため。コード側は Resend 対応済み（RESEND_API_KEY があれば
Resend を使う。cares と同じ送信サービスに揃うので、運用を新しく覚える必要はありません）。

**cares の鍵は使い回せません**: cares の Resend キーは Cloudflare Worker と GitHub Actions の
シークレット（書き込み専用）に入っていて値を取り出せず、そもそも bonds 用は別キーにするのが
安全です（bonds のキーだけ独立に無効化できる）。ドメイン認証も cares.advisers.jp とは別に必要
（**cares のドメインから営業メールを送らない**——患者向けメールの到達率を守るため）。

**手順（resend.com で一度だけ）**:
1. resend.com にログイン（cares と同じアカウント）
2. Domains → Add Domain → `bluemarl.in`（または `mail.bluemarl.in`）
   → 表示される DNS レコード（3行ほど）を bluemarl.in の DNS に貼る → Verify が緑になるまで待つ
3. API Keys → Create API Key（名前は bonds、権限は Sending access のみ）→ 値を控える
4. シークレットに投入（Cloud Shell で）:
   ```
   printf '%s' "<3のキー>" | gcloud secrets versions add BONDS_RESEND_API_KEY --data-file=- --project=arctic-anvil-497002-q2
   ```
5. デプロイ時（またはデプロイ済みなら再デプロイ時）の環境変数:
   `RESEND_API_KEY`（シークレット参照）/ `OUTREACH_FROM_EMAIL=info@bluemarl.in` /
   `OUTREACH_FROM_NAME=山口揚平` / `OUTREACH_REPLY_TO=agewaller@gmail.com`

**これで**: 受信者には「山口揚平」から届いたように見え、返信ボタンはあなたの Gmail に直行します。
（gmail.com を差出人アドレスにはできません——Gmail のドメイン認証は Google しか持てず、
名乗ると迷惑メール判定されるため。表示名＋Reply-To が業界標準の正解です）

**設定が済むまでの間も安全**: 送信だけが503で止まり、下書きの確認・修正は普通にできます。

## 4. ログインを「切れない・壊れない」にする本番設定（30分・デプロイ後）

**なぜ**: コード側は実装済み（`docs/login-reliability.md`）。ドメインが決まらないと設定できない2点だけ残っています。

**手順**:
1. `NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN` を **bonds 自身の本番ドメイン**にする（firebaseapp.com のままにしない）
2. Firebase Console → Authentication → Settings → 承認済みドメイン → bonds の本番ドメインを追加
3. GCP Console → APIとサービス → 認証情報 → OAuth 2.0 クライアント →
   「承認済みのリダイレクトURI」に `https://<bondsのドメイン>/__/auth/handler` を追加

**これで**: iPhoneのSafariでもログインが安定し、サーバCookie（14日・来訪ごと延長）で
「ログインしっぱなし」が維持されます。cares（新版）にも同じ2部品を移植できます（コードは bonds にあります）。

## 5. 返信の自動取り込み（双方向）を有効にする（30分・任意）

**なぜ**: 相手がメールに返信したら bonds のやりとりに自動で入る機能は実装済みですが、
メール受信経路（SendGrid Inbound Parse）の DNS 設定はあなたの権限が必要です。

**手順**（SendGrid 互換になるよう実装済み。設定するだけで動きます）:
1. SendGrid → Settings → Inbound Parse → Add Host & URL
2. 受信用サブドメイン（例: reply.あなたのドメイン）の MX レコードを `mx.sendgrid.net` に向ける
3. Destination URL に **シークレットを URL に付けて** 設定:
   `https://<bondsのAPI>/api/inbound/email?secret=<BONDS_INBOUND_WEBHOOK_SECRETの値>`
   （SendGrid はカスタムヘッダを付けられないため、ヘッダでなく URL クエリで認証する実装にしてあります。
   シークレットの値は `gcloud secrets versions access latest --secret=BONDS_INBOUND_WEBHOOK_SECRET --project=arctic-anvil-497002-q2` で取得）
4. これで、相手が bonds から届いたメールに返信すると、送信元アドレスで連絡先を自動判定し、
   その方の「やりとり」に返信が自動で入ります（フォーム送信・表示名つき from・envelope 経由のいずれにも対応）
