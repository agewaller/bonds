# 将来構想: ユーザーが互いに持ち寄れるマーケットプレイス

このドキュメントは、bonds の**将来の**「利用者どうしのマーケットプレイス」構想を記録し、
以降の実装がその未来を視野に入れて進むための設計指針を定める。オーナー依頼
（2026-07-18「ユーザーが互いに持ち寄れるマーケットプレイスについては今後考える。
issue に残すとともに、未来の実装を視野に入れて構築していってほしい」）に基づく。

**現状**: 単一オーナーの公開掲示板 `/market` は実装済み（下の「いま在るもの」）。
**この文書が扱うのは、その先の「複数ユーザーが互いに出品・ニーズを持ち寄り、
距離と公開設定のもとで探し合う」多者マーケットプレイス**である。まだ着手しない。

## なぜ今は掲示板止まりで、多者マーケットは先送りか

bonds は現在**単一オーナー専用**（web は OWNER_EMAIL 本人 → `ownerUid="owner"` の
単一バケツ。2026-07-18 のセキュリティ点検で匿名アクセスを塞ぎ、認証・データ分離を固めた）。
多者マーケットは bonds を**マルチテナント**へ作り替える大工事で、固めたばかりの認証・
データ分離・鍵の原則に正面から関わる。拙速にやると退行を生むため、掲示板（単一オーナーの
出品を公開し、訪問者が問い合わせ→承認で新しい連絡先になる）を先に出し、多者は別フェーズとする。

## ミッション上の意義（やる価値）

bonds のミッションは「人々の関係を最適化し、人間の共創を促進する」。多者マーケットは
**新しい関係性の構築（収集→把握→…の循環の入口を、既存の連絡先の外へ広げる）**に効く。
「誰かの申し出（give/lend/teach/help/advise）」と「誰かのニーズ」を、距離・信頼・関係の
文脈のもとで結ぶことは、bonds の「本人たちが気づかない一手」を、ネットワークの外側にも
広げる営みになる。gift（Rails EOL の P2P 譲り合い掲示板）の核はここにあった。

## いま在るもの（この上に積む・車輪の再発明をしない）

| 資産 | 場所 | 多者化で流用できる点 |
|---|---|---|
| 申し出カタログ `offerings` | `schema.prisma` / `lib/offerings.ts` | 出品の型（kind/title/desc/logistics/距離ゲート）。`published` で公開可 |
| ニーズとのマッチング `matchOfferingToContacts` | `lib/offerings.ts`（純粋関数・AI 不要・決定的） | 「申し出↔ニーズ」の照合ロジック。相手を「連絡先」から「他ユーザー」に一般化できる |
| 公開掲示板 `/market` + 問い合わせ `offering_interests` | `app.ts` 公開ルート | 公開・問い合わせ→承認で連絡先化の一周。**受け取り側の下書き→承認**の型 |
| 時間の出品 `time_offers` / 予約 `time_bookings` | `lib/time-offers.ts` | Stripe 決済つきの出品・予約。ユーザー間取引の決済経路 |
| 距離スコア 1〜5・`suggestDistance` | `lib/relationship.ts` | gift の「FB 友だちグラフの距離ゲート」に相当する、bonds 独自の距離モデル |
| やり取り台帳 `exchanges` | `lib/exchanges.ts` | 貢献/貸し借り/取引/約束の状態＋期日。ユーザー間の履歴の器 |
| 認証の uid スコープ | `lib/auth.ts` `authorizeUser` | Firebase ログインユーザーは**自分の uid スコープ**で動く実装が既にある（多者化の土台） |
| 項目暗号化 AES-256-GCM | `packages/db` | ユーザーの PII と第三者の PII を分けて守る基盤 |

**要点**: マルチテナントの土台（uid スコープ・暗号化・出品/ニーズ/距離/取引/決済）は
すでに部品として在る。足りないのは「複数ユーザーの登録」「ユーザー間の発見・照合・
安全な引き合わせ」を束ねる層と、その安全設計。

## 多者マーケットで新たに要るもの

1. **本物のマルチテナント化**
   - `ownerUid` を「単一 owner バケツ」でなく、登録ユーザーごとの Firebase uid に一般化する。
     `authorizeUser` は既に uid スコープを返せる。web の BFF・OWNER_EMAIL 特別扱いを見直し、
     一般ユーザーがそれぞれのバケツを持てるようにする（**セキュリティ点検の成果を壊さない**の最優先）。
   - 出品・ニーズ・プロフィールの**公開範囲**を、`published` の真偽値から
     `visibility`（private / contacts / public / marketplace）へ段階化する（migration で拡張）。
2. **ユーザー間の発見・マッチング**
   - `matchOfferingToContacts` を「相手＝他ユーザー」に一般化した `matchOfferingToUsers` を足す。
     ニーズ源は相手ユーザーが公開に同意した情報のみ（相手の尊厳・同意が最上位制約）。
   - **距離/信頼ゲート**: 直接の連絡先か、共通の知人（2 ホップ）か、公開かで露出範囲を変える
     （gift の距離ゲートの思想を bonds の距離モデルで実装）。むやみに全公開しない。
3. **安全な引き合わせと取引**
   - すべての外向き接触は **下書き→承認→送信**（既存原則）。ユーザー間でも、いきなり連絡先や
     本文を相手に渡さず、両者の合意で段階的に開示する。
   - 決済は既存の Stripe 経路（`time_offers`）を土台に、ユーザー間送金/手数料の是非を別途判断
     （**金融/法務の断定助言はしない**・搾取しない価格方針）。
4. **モデレーションと濫用対策**
   - 通報・ブロック・出品の審査（不適切/なりすまし/違法の排除）。第三者の PII・尊厳の保護。
   - レート制限（既存の `RateLimiter`）・スパム抑止・本人性（Firebase 認証）を土台に拡張。
5. **データ主権の維持**
   - どのユーザーも自分の出品/ニーズ/やり取りを 1 件単位で閲覧・編集・削除・エクスポートできる
     （既存原則）。相手に渡った情報の扱い（撤回・失効）も設計する。

## 採用しない（gift から引き継がない）判断は維持

- **ポイント経済**（gift の内部通貨）: 射幸性・複雑さの割に関係の質を上げない。不採用。
- **Facebook 認証/グラフ**: API 非公開・依存過多。bonds は Firebase 認証 + 独自距離モデルで代替。
- **公開の総当りマーケット（誰でも全員に見える）**: 相手の尊厳・安全と相反。露出は距離/同意で常にゲートする。

これらを再提案されたら、まずこの節の理由を確認する。

## 「未来を視野に入れて構築」— 当面の各実装で守る原則

多者マーケットに着手する前でも、日々の実装は次を守り、将来の分岐を塞がない:

1. **`ownerUid` を常にスコープキーとして扱う**（"owner" 固定を前提にした where を新たに増やさない。
   公開ルートで全 `ownerUid` を跨ぐのは掲示板が単一オーナーな今だけの妥協で、多者化時に
   per-owner スコープへ差し替える前提のコメントを残す）。
2. **公開範囲は将来 `visibility` へ拡張する含みを持たせる**（`published`/`listed` の真偽値は
   その第一歩。段階化は migration で行う。無理に今作り込まない＝YAGNI）。
3. **相手の情報の利用は同意を前提に**（公開検索・他者への開示は明示同意・下書き→承認を通す）。
4. **マッチング・距離・取引・暗号化の部品を汎用に保つ**（連絡先専用に固めず、
   「相手」を連絡先にも他ユーザーにも差せる形を意識する）。
5. **セキュリティ点検の成果（匿名アクセス封鎖・SSRF 対策・テナント分離・timingSafe）を退行させない**。
